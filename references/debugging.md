# 结构化调试指南

当用户说「卡住了」「这个 bug 搞不定」时，按以下流程执行。

---

## 第一步：收集信息（不要急着修）

向用户要求以下信息：

1. **完整错误信息** — 不是"它报错了"，而是完整的错误文本
2. **相关代码** — 错误指向的文件和行号附近的代码
3. **期望行为** — "我期望它做什么" vs "它实际做了什么"
4. **已经尝试过什么** — 避免重复无效的修复

如果用户给的信息不够，追问。信息不全就开始修 = 瞎猜。

---

## 第二步：读懂错误信息

常见错误类型速查：

| 错误类型 | 含义 | 常见原因 |
|---------|------|---------|
| `TypeError: Cannot read property 'x' of undefined` | 访问了不存在的对象的属性 | 数据还没加载就用了、拼写错误、API 返回格式不对 |
| `ReferenceError: x is not defined` | 变量不存在 | 没导入、拼写错误、作用域问题 |
| `SyntaxError` | 语法错误 | 括号不匹配、缺少逗号、JSON 格式错误 |
| `404 Not Found` | 路径不存在 | 路由配置错误、API 地址拼错 |
| `500 Internal Server Error` | 服务器端出错 | 数据库查询错误、环境变量缺失 |
| `CORS error` | 跨域请求被拒 | 后端没配置 CORS、URL 不匹配 |
| `Module not found` | 找不到模块 | 没安装依赖、导入路径错误 |

---

## 第三步：生成假设

列出 2-3 个最可能的原因，按可能性排序：

```
假设 1（最可能）：[描述] — 验证方法：[怎么验证]
假设 2：[描述] — 验证方法：[怎么验证]
假设 3：[描述] — 验证方法：[怎么验证]
```

从最可能的开始验证，不要同时改多个东西。

---

## 第四步：逐一验证

- 一次只改一个地方
- 改完立即测试
- 有效 → 问题解决
- 无效 → 撤销改动，试下一个假设

---

## 第五步：三振出局规则

**改了三次还不对 → 停下来，换思路。**

这是吴恩达在 2025 年访谈中专门强调的原则。继续在同一个方向修补，往往越改越乱。

换思路的方式：
1. **删掉重写** — 把出问题的部分删掉，用不同的方式重新实现
2. **简化问题** — 先做一个最简版本能跑，再逐步加复杂度
3. **换个工具** — 如果在 Agent 模式修不好，切到 Debug 模式或用子代理
4. **搜索文档** — AI 可能在这个问题上产生幻觉，去官方文档找正确答案
5. **隔离问题** — 创建一个最小复现案例，排除其他代码的干扰

---

## 调试时的提示词模板

**给 AI 报 bug：**
```
我遇到了这个错误：
[粘贴完整错误信息]

相关代码在 [文件路径]：
[粘贴相关代码]

我期望的行为是：[描述]
实际的行为是：[描述]

我已经尝试过：[列出已尝试的方法]
```

**让 AI 解释代码：**
```
用简单的中文解释这段代码做了什么，每个部分的作用是什么，数据是怎么流动的。
```

**让 AI 自查：**
```
审查你刚才生成的代码，找出可能的 bug、安全漏洞、和边缘情况。
```

---

## 调试后的收尾

Bug 修复后，执行 Phase 5（检查点）：
1. 确认修复有效
2. 更新 progress.txt（记录 bug 已修复）
3. 在 lessons.md 记录：什么问题 → 根因是什么 → 以后怎么避免
4. Git commit
