# 规范文档模板

生成文档时，根据用户审问结果填充以下模板。每份文档应具体、无歧义、可被 AI 直接引用。

---

## 1. PRD.md（产品需求文档）

```markdown
# [产品名称] — 产品需求文档

## 产品概述
一句话描述：[什么产品，给谁用，解决什么问题]

## 目标用户
- 主要用户：[描述]
- 使用场景：[何时何地使用]

## 核心功能（MVP）
1. [功能名] — [一句话描述] — 成功标准：[可验证的标准]
2. [功能名] — [一句话描述] — 成功标准：[可验证的标准]
3. [功能名] — [一句话描述] — 成功标准：[可验证的标准]

## 用户故事
- 作为[角色]，我想要[行动]，以便[目的]
- ...

## 非目标（明确不做）
- [功能/特性] — 原因：[为什么不做]
- ...

## 成功指标
- [指标1]
- [指标2]
```

## 6. IMPLEMENTATION_PLAN.md（实施计划）

```markdown
# [产品名称] — 实施计划

## Phase 1: 项目初始化
- [ ] 1.1 初始化项目，安装 TECH_STACK.md 中的依赖
- [ ] 1.2 创建目录结构
- [ ] 1.3 配置环境变量（.env）
- [ ] 1.4 初始化 Git

## Phase 2: 基础框架
- [ ] 2.1 按 FRONTEND_GUIDELINES.md 创建全局样式/主题
- [ ] 2.2 创建布局组件（导航栏、页脚、侧边栏）
- [ ] 2.3 配置路由（按 APP_FLOW.md 的页面清单）

## Phase 3: 核心功能
- [ ] 3.1 [功能名] — [具体要做什么]
- [ ] 3.2 [功能名] — [具体要做什么]
- [ ] 3.3 [功能名] — [具体要做什么]

## Phase 4: 数据层
- [ ] 4.1 按 BACKEND_STRUCTURE.md 创建数据库表
- [ ] 4.2 实现 API 端点
- [ ] 4.3 连接前端到 API

## Phase 5: 认证与安全
- [ ] 5.1 集成认证服务
- [ ] 5.2 添加路由保护
- [ ] 5.3 验证 .env 密钥不暴露

## Phase 6: 打磨与发布
- [ ] 6.1 响应式测试（手机、平板、桌面）
- [ ] 6.2 空状态和错误状态处理
- [ ] 6.3 加载状态
- [ ] 6.4 部署

---
每个步骤应足够具体，让 AI 不需要猜测。
提示词模式："构建 IMPLEMENTATION_PLAN.md 的步骤 3.2"
```

---

## 生成原则

- 每份文档必须具体到可以被 AI 直接引用执行
- 文档之间相互引用：PRD 定义功能 → APP_FLOW 定义体验 → TECH_STACK 定义工具 → FRONTEND_GUIDELINES 定义外观 → BACKEND_STRUCTURE 定义数据 → IMPLEMENTATION_PLAN 定义顺序
- 宁可过于具体，不可模糊。模糊 = AI 猜测 = 幻觉
- 用户审问中没有回答的部分，标记为 `[待确认]`，不要自行假设
