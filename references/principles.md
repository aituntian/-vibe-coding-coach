# Vibe Coding 核心原则与常见坑

综合全球和中国社区的实战经验，提炼出的核心原则和常见陷阱。

---

## 十大核心原则

### 1. 文档第一，代码第二
在写任何代码之前，先写清楚你要做什么。6 份规范文档 + 3 份会话文件 = AI 不需要猜测。没有文档的 Vibe Coding 就是在祈祷。

### 2. 一次只改一件事
大请求 = AI 猜测 = 幻觉 = 屎山。把大功能拆成小步骤，每步都能看懂、能验证。"构建 IMPLEMENTATION_PLAN.md 的步骤 3.2" 永远好过 "构建整个登录系统"。

### 3. 先讨论方案再执行
Ask/Plan 模式先行，Agent 模式后行。先让 AI 告诉你它打算怎么做，你审核同意了再让它动手。这能避免 AI 走错方向后你花更多时间修。

### 4. 每 15-20 分钟 commit 一次
Git 是你的救命稻草。每完成一个可运行的小功能就 commit。出问题可以回滚。不 commit 就改下一个功能 = 在钢丝上走路不系安全绳。

### 5. 三次不对就重来
吴恩达原则：同一个 bug 修了三次还不对，删掉换思路重来。继续在错误方向上修补只会越改越乱。

### 6. 具体打败模糊
"用 #3B82F6 做背景，16px 内边距，8px 圆角" 每次都打败 "做成蓝色加点间距"。你的提示词越具体，AI 猜测越少，输出越好。

### 7. 截图胜过千言
UI 工作时，喂截图给 AI 比文字描述好 10 倍。找到你喜欢的设计，截图，说"匹配这个布局"。

### 8. CLAUDE.md 是活文档
每次 AI 犯错你纠正它时，更新 CLAUDE.md 或 lessons.md。随着时间推移，AI 的错误率可测量地下降。这是自我改进循环。

### 9. MVP 先行
砍到 3-5 个核心功能，做完再加。功能膨胀杀死的项目比坏代码多得多。你从未发布的完美 App 比已上线的普通 App 更糟。

### 10. 审查每一行 diff
不要盲目 Accept All。用 `git diff` 审查 AI 生成的每次变更。AI 可能引入不存在的 API、错误的逻辑、安全漏洞。信任但验证。

## 十大常见坑

### 坑 1：需求膨胀
"加功能太容易了" → PRD 从 V1 到 V_Final_Ultra → 项目失控。
**解法：** MVP 先行，PRD.md 中明确写「非目标」。

### 坑 2：一次让 AI 生成太多代码
AI 生成的代码量越大，越难理解和掌控，出错后排查成本指数级上升。
**解法：** 拆成小步骤。"先做数据结构 → 再做列表渲染 → 最后加样式"。

### 坑 3：盲目信任 AI 输出
AI 可能生成不存在的 API、错误的逻辑、有安全漏洞的代码。
**解法：** 每次生成后审查和测试，不要直接 Accept All。

### 坑 4：上下文丢失（多莉综合征）
对话轮次多了，AI 会"忘记"最初的目标和约束，代码开始偏离方向。
**解法：** 新话题开新对话；用 CLAUDE.md 固定项目上下文；用 `/compact` 压缩长对话。

### 坑 5：反复修同一个 bug
与 AI 无效拉扯，越改越乱。
**解法：** 三振出局 — 改了三次还不对就删掉换思路。

### 坑 6：代码变成屎山
AI 倾向于生成冗余代码、过度抽象、硬编码值。
**解法：** 定期要求 AI 重构和精简；关键逻辑自己审查。

### 坑 7：文档漂移
AI 改了代码但不同步更新注释和文档。
**解法：** 每次功能完成后，明确要求 AI 更新相关文档。

### 坑 8：过早优化和过度抽象
AI 喜欢搞"最佳实践"，但项目初期需要的是能跑起来。
**解法：** 奥卡姆剃刀 — 如无必要，勿增代码。三行相似代码好过一个过早的抽象。

### 坑 9：忽视版本控制
不提交就改下一个功能，出问题后无法回滚。
**解法：** 每完成一个可运行阶段就 git commit。

### 坑 10：以为零代码 = 零门槛
Vibe Coding 不需要你写代码，但需要你会拆解需求、描述问题、评估结果。
**解法：** 学会当"产品经理"而不是"程序员"。学会拆解需求比学会写代码更重要。

## 提示词模式速查

| 场景 | 差的提示词 | 好的提示词 |
|------|-----------|-----------|
| 开始功能 | "给我做个登录" | "构建 IMPLEMENTATION_PLAN.md 的步骤 5.1。登录流程在 APP_FLOW.md 第 3 节。使用 BACKEND_STRUCTURE.md 的认证方案。按 FRONTEND_GUIDELINES.md 样式化。" |
| UI 调整 | "让它好看点" | "卡片用 #F9FAFB 背景、16px 内边距、8px 圆角、1px #E5E7EB 边框。参考附图的布局。" |
| 修 Bug | "它坏了修一下" | "我遇到这个错误：[完整错误]。代码在 [文件:行号]。期望行为是 [X]，实际是 [Y]。" |
| 新会话 | "我们继续" | "先读取 CLAUDE.md、progress.txt 和 lessons.md。告诉我当前状态和建议的下一步。" |
| 防止偏离 | （不说） | "只构建 PRD.md 中的功能。不要添加我没要求的东西。" |
| 防止乱装包 | （不说） | "只使用 TECH_STACK.md 中的包。引入新依赖前先问我。" |

## 上下文管理技巧

- **CLAUDE.md 保持精简** — 60 行以内最佳，不超过 300 行。太长反而降低 AI 遵循度
- **用指针不用内容** — CLAUDE.md 中写"参考 FRONTEND_GUIDELINES.md 的调色板"，不要把调色板复制进来
- **关键信息放开头或结尾** — AI 对中间位置的信息注意力较弱（lost-in-the-middle 问题）
- **新话题开新对话** — 上下文窗口是有限资源，不要在一个对话里做所有事
- **长对话开新会话** — 上下文窗口是有限资源，话题切换时开新对话，用 CLAUDE.md 保持连续性
